## Баг: asyncio event loop isolation в embedder.py

### Симптом

```
WARNING  Embedding unavailable — falling back to BM25-only
```

Embeddings не работали при запуске через MCP-сервер даже с корректным `OPENAI_API_KEY`.

### Причина

`embedder.py` содержал module-level объекты:

```python
_HTTP: httpx.AsyncClient | None = None
_SEM = asyncio.Semaphore(1)
```

`asyncio.Semaphore` привязывается к event loop, в котором был создан.
MCP-сервер (FastMCP) работает в loop L1.
`KnowledgeBase._embed_for_recall` вызывает `asyncio.run(embed_texts(...))` через `ThreadPoolExecutor` — создаётся новый loop L2.
Попытка использовать `_SEM` из L1 внутри L2:

```
RuntimeError: Task <...> got Future <asyncio.locks.Semaphore ...> attached to a different loop
```

Исключение ловилось `except Exception` → логировалось как "Embedding unavailable" → silent fallback на BM25.

### Фикс

Убраны оба синглтона. Каждый вызов `embed_texts` создаёт свой `AsyncClient` как context manager:

```python
async with httpx.AsyncClient(timeout=timeout) as client:
    resp = await client.post(...)
```

Функция теперь безопасна для вызова из любого event loop.

## Конфигурация для запуска бенчмарка с OpenAI

```env
OPENAI_API_KEY=sk-...
DEFAULT_JUDGE_MODEL=gpt-4o-mini
DEFAULT_ANSWER_MODEL=gpt-4o
AI_KNOT_PROVIDER=openai
AI_KNOT_MODEL=gpt-4o-mini
AI_KNOT_EMBED_URL=https://api.openai.com
AI_KNOT_EMBED_MODEL=text-embedding-3-small
# AI_KNOT_EMBED_API_KEY не нужен — mcp_server подхватит OPENAI_API_KEY
```

Запуск через `bun run` (автозагрузка `.env`):

```bash
cd aiknotbench && bun run src/index.ts run -r v094-openai \
  --ingest-mode dated-learn --top-k 60 --types 1,2,3,4
```
